2 ЗАВДАННЯ
        На лабораторній роботі №2 потрібно розробити базу даних для серверної частини  програмної системи та прикладного програмного інтерфейсу. 
Завдання:
1. Розробити будову програмної системи.
2. Створити UML діаграму прецедентів для серверної частини системи.
3. Створити ER діаграму даних.
4. Розробити базу даних (БД) програмної системи.
5. Створити діаграму структури БД.
6. Розробити функції роботи з БД (ORM або CoRM тощо).
7. Розробити API (REST або GraphQL, gRPC тощо) для взаємодії серверної частини з клієнтами.
8. Створити специфікацію розробленого API.
9. Створити програмну реалізацію розробленого API та функцій роботи з БД.
10. Перевірити роботу створеного програмного коду серверної частини системи.






3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Архітектура серверної частини системи


Серверна частина системи реалізована на основі клієнт-серверної архітектури. Взаємодія між клієнтами та сервером здійснюється за допомогою REST API. Сервер відповідає за автентифікацію користувачів і пристроїв, обробку сенсорних даних, виконання бізнес-логіки, генерацію сповіщень та збереження інформації у базі даних.
Для зберігання даних використовується реляційна система управління базами даних PostgreSQL. Серверна частина та база даних розгортаються у Docker-контейнерах, що забезпечує ізоляцію середовища та зручність розгортання.


3.2 UML діаграма прецедентів (Use Case Diagram)
UML діаграма прецедентів використовується для опису функціональних можливостей серверної частини системи та взаємодії користувачів з нею.
У системі визначено три основні актори:
* User — зареєстрований користувач системи;
* Admin — користувач з розширеними адміністративними правами;
* IoT Device — пристрій збору сенсорних даних.


Користувач може виконувати автентифікацію, керувати пристроями, переглядати сенсорні показники, працювати з підписками та отримувати сповіщення. Адміністратор має додаткові можливості аналітики та контролю системи, включаючи перегляд журналів подій і розрахунок аналітичних метрик. IoT-пристрої автономно надсилають сенсорні дані та інформацію про свій стан.
Для моделювання складної поведінки системи використано відношення <<include>> та <<extend>>, що дозволяє розділити обов’язкову та опціональну функціональність. Рисунок Б.1 – UML діаграма прецедентів серверної частини системи.


3.3 ER діаграма даних


ER діаграма використовується для наочного відображення структури бази даних та зв’язків між сутностями.
Основними сутностями бази даних є:
* користувачі системи;
* пристрої збору даних;
* локації розміщення пристроїв;
* сенсорні показники;
* алерти;
* підписки;
* журнали аудиту;
* обчислені аналітичні дані;
* інформація про стан пристроїв.


Між сутностями реалізовані зв’язки типу «один-до-багатьох», що дозволяє одному користувачу володіти кількома пристроями, одному пристрою мати множину вимірювань та алертів, а також зберігати історію обчислених показників. Рисунок Б.2 – ER діаграма бази даних системи.


3.4 Діаграма структури бази даних


Діаграма структури бази даних деталізує таблиці, їх атрибути, первинні та зовнішні ключі, а також обмеження цілісності.
Для кожної таблиці визначено:
* первинний ключ;
* зовнішні ключі;
* типи даних;
* обмеження унікальності та зв’язків.


Структура бази даних відповідає принципам нормалізації (1NF–3NF), що забезпечує мінімізацію надлишковості даних та цілісність інформації. Рисунок Б.3 – Діаграма структури бази даних.


3.5 Розробка API серверної частини


Для взаємодії клієнтів із серверною частиною було розроблено REST API. API підтримує CRUD-операції для основних сутностей системи, включаючи користувачів, пристрої, сенсорні показники, алерти та підписки.        
Для тестування API використовувалися інструменти Postman та Swagger, що дозволило перевірити коректність роботи серверної частини та взаємодії з базою даних.






4 ВИСНОВКИ
У ході виконання лабораторної роботи №2 було розроблено архітектуру серверної частини програмної системи моніторингу радіаційного фону. Було спроєктовано та реалізовано структуру бази даних, створено UML та ER діаграми, а також визначено API для взаємодії з клієнтами та IoT-пристроями.
Отримані результати дозволили сформувати цілісне уявлення про процес проєктування серверної частини програмної системи, принципи побудови баз даних, використання UML-діаграм та розробку API. Реалізоване рішення є основою для подальшої розробки бізнес-логіки та функцій адміністрування у наступних лабораторних роботах.
